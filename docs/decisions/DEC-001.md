# DEC-001: KAP Edge Cases & Holiday Handling

---

## CONTEXT

KAP (Kurumsal Yönetim Disclosure) API is critical Layer 3 data source for corporate announcements and ownership changes. However, it experiences:
1. **Holiday closures** (weekends, national holidays – no announcements published)
2. **WAF blocks** (rate limiting after 500+ requests – temporary 403 errors)
3. **Bulk announcement spikes** (earnings season – queue processing delays)

Previous behavior: Naive fetch → error → cascade to cache (losing same-day data window)

**Problem:** System couldn't distinguish between "no announcements" (normal) vs "API down" (abnormal). Wasted fallback attempts.

---

## OPTIONS CONSIDERED

### Option A: Treat All KAP Errors as Data Missing
- Fetch fails → immediately cascade to ETF/sentiment proxies
- **Pros:** Simple logic
- **Cons:** Lose legitimate KAP data on holidays (expected behavior)
- **Cons:** No recovery path for WAF blocks (need retry after 5min)

### Option B: Add Holiday Calendar (CHOSEN)
- Detect BIST closure before fetch
- If closed: Skip fetch, use cache (avoid unnecessary API calls)
- If open: Fetch → validate → handle WAF blocks with exponential backoff
- If WAF blocks: Queue request for 15min later, use cache meanwhile
- **Pros:** Respects market calendar
- **Pros:** 3-tier fallback reduces wasted calls
- **Cons:** Requires maintaining holiday calendar

### Option C: Always Retry with Backoff
- Fetch fails → wait 5min → retry
- **Pros:** Recovers from temporary WAF blocks
- **Cons:** Blocks daily workflow (unacceptable latency)
- **Cons:** No holiday awareness (still wastes calls)

---

## DECISION: Option B (Holiday-Aware with Fallback)

**Rationale:**
1. **Market efficiency:** Skip API calls on known non-trading days
2. **Reliability:** 3-tier handling (success → fallback → cache)
3. **Observability:** Clear distinction between normal (holiday) and abnormal (WAF) failures
4. **Testing:** Holiday calendar + bulk queue + cache = comprehensive edge case coverage

## IMPLEMENTATION

### Architecture Changes

**File: `src/data/kap.py`**

```python
from config import BIST_HOLIDAY_CALENDAR

class KAPFetcher:
    def fetch_announcements(self, ticker, date=None):
        """
        Holiday-aware KAP fetch with WAF resilience.
        """
        date = date or datetime.today()
        
        # Step 1: Holiday check
        if self.is_bist_closed(date):
            logger.info(f"BIST closed on {date}, skipping KAP fetch")
            return self.fetch_from_cache(ticker)  # Use yesterday's cache
        
        # Step 2: Try fetch with queue
        try:
            announcements = self.kap_api.get_announcements(ticker, date)
            self.update_cache(ticker, announcements)
            return announcements
        except HTTPError(429):  # Rate limit / WAF block
            logger.warning(f"KAP WAF block detected, queuing retry")
            self.queue_retry(ticker, delay_minutes=15)
            return self.fetch_from_cache(ticker)  # Use cache during retry
        except Exception as e:
            logger.error(f"KAP fetch failed: {e}")
            return self.fetch_from_cache(ticker)
    
    def is_bist_closed(self, date):
        """Check BIST holiday calendar."""
        return date in BIST_HOLIDAY_CALENDAR or date.weekday() >= 5  # Saturday/Sunday
    
    def queue_retry(self, ticker, delay_minutes=15):
        """Queue failed request for later retry."""
        self.retry_queue.put({
            'ticker': ticker,
            'retry_at': datetime.now() + timedelta(minutes=delay_minutes)
        })
```

## INTEGRATION POINTS

### Where This Changes Things

**Before (Naive Fetch):**
```
ticker → fetch_kap() → error? → cache → done
```

**After (Holiday-Aware with Queue):**
```
ticker → is_bist_closed()? 
  → YES → use cache (skip API call) →
  → NO → fetch_kap()
       → Success → cache + return →
       → WAF 429 → queue_retry() + use cache →
       → Other error → cache + log →
```

### Configuration Impact

New settings in `config.yaml`:
- `kap.holidays` (array of dates)
- `kap.retry_delay_minutes` (15 default)
- `kap.queue_enabled` (true default)
- `kap.cache_ttl_incident_hours` (72 hours emergency mode)

## RISKS & MITIGATIONS

| Risk | Severity | Mitigation |
|---|---|---|
| **Holiday calendar outdated** | Medium | Quarterly KAP update, alerts if holiday not in calendar |
| **Queue corruption on crash** | Low | Atomic writes, recovery on restart |
| **Memory leak in retry queue** | Low | Max queue size limit (100), old items expire |
| **WAF block persistence** | Low | Exponential backoff (15→30→60 min), manual override |
| **Concurrent queue access** | Low | Lock-based queue (thread-safe) |
